# Task Plan

[作業]タスクの実装計画を立て、PR分割と依存関係を整理するスキル。
plan.md作成までが責務。実装はtask-executeが担う。

## トリガー条件

以下のすべてを満たす場合に使用する：
- cc-memoryに`[作業]`タスクがある
- ユーザーが実装を依頼した
- plan.mdがまだ存在しない

以下では使用しない：
- plan.mdが既に存在する → task-execute を使う
- 仕様が未確定 → discussion / design スキルを使う
- 調査・検索・質問応答 → 本体が直接対応

## ワークフロー

### Step 0: 仕様確認

メインエージェントが以下を実行する（SAには委譲しない）：

1. **決定事項の取得**: 関連トピックの `get_decisions` を取得する
2. **設計書・仕様書の確認**: 参照すべきドキュメントの存在確認と読み込み
3. **ユーザーへの仕様確認**: 決定事項と設計書の内容をユーザーに提示し、以下を確認する
   - 実装スコープが正しいか
   - 変更すべき仕様はないか
   - 追加の要件はないか
4. **ユーザーのGOが出てから** Step 1 に進む

**ユーザーへの提示テンプレート：**

```
## 仕様確認

### 関連する決定事項
- id:{N}: {決定内容の要約}

### 設計書の要点
- {設計書から抜粋した重要ポイント}

### 実装スコープ
- {何を実装するか}
- {何を実装しないか}

### 確認事項
- {不明点や確認したいこと}

→ この内容で進めてOK？
```

### Step 1: 変更内容の分類

タスクの変更内容を **PR分割基準（5類型）** で分類する。
レビューの「読み方」が同じものだけ1 PRにまとめる。異なる類型を1 PRに混ぜない。

| 類型 | 内容 | レビューの読み方 |
|------|------|----------------|
| A. 機械的置換 | リネーム、import書き換え | 差分を流し読みで確認 |
| B. スキーマ/データ変更 | テーブル追加/削除、トリガー、マイグレーション | データモデルの妥当性を精査 |
| C. 新規モジュール追加 | 新ファイル、新クラス | ゼロからの設計レビュー |
| D. 既存コードの振る舞い変更 | ロジック変更、API変更 | before/afterの差分を精査 |
| E. 削除/廃止 | 旧コード除去 | 参照残りがないか確認 |

### Step 2: ブランチ戦略の判断

PRが複数になる場合、全PRをmainに直接向けるか、統合ブランチ（feature branch）を経由するかを判断する。

- **main直撃（stacked PRs）**: 各PRが独立してmainに入っても安全な場合
- **統合ブランチ経由**: 中間状態でmainが壊れる、または機能として一体で部分マージに意味がない場合

統合ブランチを選択した場合：
- PR分割テーブルの末尾に `final`（統合マージ）行を追加する
- 各サブPRのbaseは統合ブランチとし、finalのbaseはmainとする
- finalの依存先は他の全サブプラン

### Step 3: plan.md 作成

出力先: `~/.claude/projects/<project>/work/{task-name}/`

**PR が1個で済む場合**: plan.md 単体（サブプランは作らない）

**PR が複数になる場合**: plan.md（インデックス）+ サブプラン（plan-a.md, plan-b.md, ...）に分割。
各サブプラン = 1 PR = 1 SA実行単位。

#### plan.md テンプレート（複数PR）

```markdown
# {タスク名}

## 作業背景
{なぜこのタスクが必要か}

## 関連情報
- cc-memory タスク: #{タスクID}
- 関連 decisions: #{ID}, #{ID}, ...
- 設計書: {パス}（あれば）

## ブランチ戦略
{main直撃 or 統合ブランチ（feature/{統合ブランチ名}）}

## PR分割

| # | 類型 | 内容 | ブランチ名 | base | 状態 |
|---|------|------|-----------|------|------|
| a | {A-E} | {概要} | feature/{名前} | main | 🔲未着手 |
| b | {A-E} | {概要} | feature/{名前} | main or PR-aのブランチ | 🔲未着手 |
| c | {A-E} | {概要} | feature/{名前} | main or 依存先 | 🔲未着手 |
| final | - | 統合マージ | feature/{統合ブランチ名} | main | 🔲未着手 |

状態: 🔲未着手 → 🔄着手中 → ✅完了
※ final行は統合ブランチ戦略の場合のみ。main直撃の場合は不要。

## 依存関係
- PR-b は PR-a の完了後に着手（{理由}）
- PR-c は独立して着手可能
- final は全サブプランの完了後（統合ブランチ戦略の場合）

## サブプラン
- [plan-a.md](./plan-a.md): {概要}
- [plan-b.md](./plan-b.md): {概要}
- [plan-c.md](./plan-c.md): {概要}
```

#### サブプラン テンプレート

```markdown

## 類型
{A-E のどれか}

## ブランチ
- ブランチ名: feature/{名前}
- base: {main or 依存先ブランチ}

## TODOリスト
- [ ] TODO 1: {内容}
- [ ] TODO 2: {内容}
- ...

## 仕様参照
- 設計書: {パス}
- 参考実装: {パス}
- テスト: {パス}

## 実装順序
{TODOの推奨実装順序と依存関係}

## 実装パターン
{既存コードから抽出したパターンの具体例。コードスニペット付き}

## 受け入れ基準
- [ ] {基準}
- [ ] テストが追加されている
- [ ] 既存テストが通る

## テスト実行方法
{テストの実行コマンド}
```

### Step 4: ユーザー確認

plan.md（とサブプラン）を作成したら、**必ずユーザーに確認を取る**。

提示する内容：
- PR分割テーブル（何をどう分けたか）
- 依存関係（どの順で進めるか）
- ブランチ名とbase branch
- plan.md の絶対パス

**実装には絶対に進まない。** ユーザーが確認してOKを出すまで待つ。

## 注意事項

- plan.md作成前にファイル操作を始めない
- ブランチ名やbase branchを勝手に決めない（plan.mdに書いてユーザーに確認）
- 既存コードのパターンを調査してplan.mdに記載する（SAが参照できるように）
- 仕様が不明確な場合はユーザーに確認（勝手に解釈しない）
